## 1. 設計方針

### 1.1 アーキテクチャの基本姿勢

- **シンプル優先**: MVPは「動くものを最短で出す」ことを優先。過度な抽象化や複雑な設計パターンは避ける
- **純粋関数中心**: ゲームのロジックは入力→出力が明確な純粋関数として実装し、テスト可能性を担保する
- **静的データの宣言性**: 政策・イベント・トレンド等は型付きの宣言的データとして外部化し、コードと分離する
- **クライアント完結**: MVPはサーバーを持たず、すべてブラウザ内で完結する
- **将来拡張への余白**: 設計判断は将来の機能追加 (教育サポート, セーブ・ロード, マルチプレイ等) を阻害しないように行う

### 1.2 採用しないアプローチ

- **過度な状態管理ライブラリ**: Redux 等の重厚なライブラリは MVP には不要
- **サーバーサイドレンダリング**: シングルプレイのゲームでは恩恵が薄く、開発複雑度を増すだけ
- **詳細なドメイン駆動設計**: ゲーム規模に対してオーバーキル

## 2. 技術スタック

### 2.1 採用構成

| 領域 | 採用技術 |
|---|---|
| ビルドツール | Vite |
| UIフレームワーク | React 19+ |
| 言語 | TypeScript |
| 状態管理 | Zustand |
| スタイリング | Tailwind CSS |
| コンポーネントライブラリ | shadcn/ui (Radix UI ベース) |
| ルーティング | (MVP不要, 将来 TanStack Router を検討) |
| テスト | Vitest + React Testing Library |
| Linter/Formatter | Biome |

### 2.2 Next.js を採用しない理由 (MVP段階で)

要件定義書段階では Next.js も候補だったが、MVP では以下の理由で Vite を採用する。

- 本ゲームはクライアント完結の SPA であり、Next.js の主要機能 (SSR, API Routes, 画像最適化等) を活用する場面が少ない
- App Router の "use client" ディレクティブを多用することになり、Next.js の旨味が出ない
- Vite の方が起動・ビルド速度が速く、開発体験が良い
- 静的ビルドして Netlify 等に簡単にデプロイ可能

ただし、将来サーバー機能 (リーダーボード, セーブ同期, マルチプレイ等) を追加する段階では、Next.js への移行も検討に値する。Vite から Next.js への移行コストは限定的なので、現時点で Vite を選択しても将来詰む選択ではない。

| 用途 | ライブラリ |

| 状態管理 | zustand |
| イミュータブル更新 | immer (zustand と統合) |
| UIユーティリティ | clsx (条件付きクラス名) |
| ID生成 (任意) | nanoid |

## 3. アプリケーション全体構成

### 3.1 レイヤー構成

本アプリケーションは以下の4層構造で構成する。

```
┌─────────────────────────────────────────────┐
│ UI Layer                                    │
│   React Components (画面・コンポーネント)   │
│   ─────────────────────────────────────     │
│   依存: State Layer, Data Layer             │
├─────────────────────────────────────────────┤
│ State Layer                                 │
│   Zustand Stores (gameStore, uiStore)       │
│   ─────────────────────────────────────     │
│   依存: Logic Layer, Data Layer             │
├─────────────────────────────────────────────┤
│ Logic Layer (Game Engine)                   │
│   純粋関数 (政策判定, 選挙計算, スコア計算) │
│   ─────────────────────────────────────     │
│   依存: Data Layer                          │
├─────────────────────────────────────────────┤
│ Data Layer                                  │
│   静的データ (政策・イベント・トレンド等)   │
│   ─────────────────────────────────────     │
│   依存: なし                                │
└─────────────────────────────────────────────┘
```

下位レイヤーは上位レイヤーに依存しない。これによりテスト容易性と保守性を確保する。

### 3.2 各レイヤーの責務

| レイヤー | 責務 | 含まれるもの |
|---|---|---|
| UI Layer | ユーザーへの表示と操作受け付け | 画面コンポーネント, ダイアログ, 共通UI |
| State Layer | アプリケーション状態の保持と更新 | gameStore, uiStore |
| Logic Layer | ゲームの計算・状態遷移ロジック | 政策通過判定, 選挙計算, ターン進行 |
| Data Layer | 静的データ定義 | 政策48本, イベント50本, 政党, 地方等 |

## 5. 状態管理とデータフロー

### 5.1 状態の分類

| 状態 | 配置 | 説明 |
|---|---|---|
| ゲーム状態 | `gameStore` | GameState (全ゲームデータ) |
| UI状態 | `uiStore` | 現在画面ID, 開いているモーダル, 一時選択など |
| 静的データ | レジストリ (起動時に読み込み) | 政策・イベント等の定義 |
| コンポーネントローカル状態 | React useState | フォーム入力中の値など |

### 5.2 状態管理の方針

- **Single Source of Truth**: ゲーム状態は `gameStore` に集約。コンポーネントが個別に保持しない
- **イミュータブル更新**: Zustand + Immer で「ミュータブルに書けるが実体はイミュータブル」な更新を行う
- **アクション中心**: 状態の変更はすべて Store のメソッド (アクション) を経由
- **派生値はセレクタ経由**: 計算が必要な値はストアのセレクタで取得

### 5.3 データフロー

```
[ユーザー操作]
       ↓
[UI コンポーネント]
       ↓ (アクション呼び出し)
[Store のアクション]
       ↓ (engine の純粋関数を呼ぶ)
[Logic Layer の純粋関数]
       ↓ (新しい state を返す)
[Store が state を更新]
       ↓ (購読中の UI に通知)
[UI コンポーネントが再描画]
```

### 5.4 Store の責務分担

**gameStore**:

- GameState の保持と更新
- アクション (政策提出, イベント対応, 予算配分, ターン進行等)
- engine の純粋関数を呼び出して新しい state を作る

**uiStore**:

- 現在表示中の画面ID
- 開いているモーダルとそのパラメータ
- 一時的なフォーム入力 (確定前)
- 設定 (フォントサイズ等, 将来)

## 6. ゲームエンジン (計算ロジック層)

### 6.1 純粋関数の原則

ゲームエンジンの関数はすべて以下を満たす。

- 引数のみから返り値を導出する (副作用なし)
- 同じ入力に対して常に同じ出力 (決定的)
- 引数を破壊しない (イミュータブル)

これにより、ユニットテストが容易になり、状態のリプレイ・デバッグもしやすい。

### 6.2 主要モジュール

engine/ 配下にはドメインごとにモジュールを配置する。各モジュールは関連する純粋関数群をまとめる単位として機能する。MVPで必要となる主要なドメインは以下の通り。

- イデオロギー関連の計算 (乖離・距離)
- 政策通過判定とその効果適用
- 予算処理 (本予算・補正予算)
- 選挙計算 (議席算出)
- 連立形成と連立健全度の更新
- イベント発生判定と効果適用
- トレンド進行
- 最終スコアとエンディング判定
- ターン進行のオーケストレーション

各ドメインごとの計算式の詳細は機能設計書を参照。

### 6.3 ターン進行の流れ

ターン進行のオーケストレーションモジュールがフェーズの順序を制御する。各フェーズで該当するドメインの純粋関数を呼び出し、新しい状態を返す。

フェーズの順序:

1. イベント発生と対応
2. 政策提出と通過判定
3. 予算配分 (本予算 or 補正予算)
4. 党運営 (自動処理)
5. 指標更新 (トレンド進行 + 効果集計)
6. 選挙 (該当ターンのみ) → 連立交渉
7. 履歴保存と終了判定

各フェーズの具体的な処理内容は機能設計書を参照。

## 7. 静的データの管理

### 7.1 データの構造化方針

静的データはカテゴリ別にファイルを分割し、`data/registry.ts` で集約する。

```
data/policies/tax-finance.ts:
  → 税制財政カテゴリの政策 8本を Policy[] としてエクスポート

data/registry.ts:
  → 全カテゴリの政策・イベント・トレンド等を集約し、
    GameDataRegistry として export
```

### 7.2 計算係数 (バランスデータ)

ゲームバランス設計書で定義される計算係数は `data/balance.ts` に集約する。

例:

```
data/balance.ts:
  - 選挙ウェイト計算の与党係数 / 連立党係数 / 野党係数
  - トレンド進行の抑制係数
  - 各政策の効果値
  - 経済情勢補正のスケール
```

これらを engine の純粋関数が参照する形にすることで、バランス調整時にコードを変更せずに数値だけ調整できる。

### 7.3 シナリオ

シナリオは「初期 GameState を生成する関数」として実装する。MVPは標準シナリオ1つだが、将来は複数追加可能。

```
data/scenarios/standard.ts:
  export const standardScenario: Scenario = {
    id: 'standard',
    name: '標準シナリオ',
    description: '...',
    createInitialState: (config) => { ... }
  }
```

## 8. UI層の設計

### 8.1 コンポーネントの分類

| 分類 | 配置 |
|---|---|
| 画面コンポーネント | `components/screens/` |
| ダイアログ | `components/dialogs/` |
| 共通レイアウト | `components/common/` |
| 汎用UI部品 | `components/ui/` |

汎用UI部品 (Button, Dialog, Slider, Tabs, Progress 等) は **shadcn/ui** を使用する。shadcn/ui はライブラリではなく、CLI でコンポーネントのソースコードを `components/ui/` にコピーする方式のため、必要なコンポーネントのみを段階的に取り込める。また、コピーされたコードは自プロジェクトの一部となるため、ゲーム固有のスタイル調整も自由に行える。

ダイアログ (`components/dialogs/`) はゲーム固有のダイアログ (解散総選挙の確認等) を実装する場所で、内部的に shadcn/ui の Dialog を利用する。

### 8.2 画面コンポーネントの基本構造

各画面コンポーネントは以下のパターンに従う。

1. `useGameState` 等のカスタムフックで必要な状態を取得
2. 表示用の派生値をローカルで計算 (重い計算は engine 経由)
3. ユーザーアクションは `gameStore` または `uiStore` のアクションを呼ぶ
4. 共通レイアウト (Header/StatusBar/Footer) はLayout コンポーネントで包む

### 8.3 共通レイアウト

全画面で使用する共通要素 (ヘッダー・ステータスバー・フッター) は共通レイアウトコンポーネントで一元管理する。画面種別に応じてレイアウトを切り替える。

- ゲーム中の画面: ヘッダー・ステータスバー・フッターをすべて表示
- タイトル画面・党設定画面: 最小レイアウト (ヘッダー類を非表示)
- エンディング画面: 専用レイアウト

### 8.4 説明ボックス

各画面のメインエリア最上部に表示する「説明ボックス」 (UI/画面設計書 §2.4 参照) は、`components/common/` に再利用可能なコンポーネントとして配置する。タイトル・本文を props で受け取り、画面コンポーネント側がメインエリアの最上部に配置する。用語ツールチップは別の共通コンポーネント (例: `InfoTooltip`) として実装し、説明ボックスや画面本体から自由に組み込めるようにする。

### 8.5 カスタムフック

頻繁にアクセスする派生値はカスタムフックでカプセル化する。例として以下のような値:

- ゲーム状態全体への参照
- プレイヤー党の情報
- 現在のフェーズ
- 5指標の値
- 現在表示中の画面ID

これらをフック化することで、各コンポーネントから簡潔にストアの値を参照できるようになる。

## 9. ルーティング

### 9.1 MVPでの方針: ルーターなし (状態駆動)

MVPでは React Router や TanStack Router を使用しない。代わりに `uiStore.currentScreen` に基づく状態駆動の画面切り替えを行う。

```
App.tsx の擬似構造:
  switch (currentScreen):
    case 'title':       <TitleScreen />
    case 'party_setup': <PartySetupScreen />
    case 'dashboard':   <DashboardScreen />
    case 'event':       <EventScreen />
    ...
```

理由:

- URL を介した状態共有はゲームでは不要 (セーブ・ロード時に状態は別途復元)
- ブラウザ戻るボタンでターンを巻き戻されると整合性が崩れる
- 実装が単純

### 9.2 将来のルーティング導入

以下の場合にルーターの導入を検討する。

- ヘルプ・用語集・ルール画面を独立したURLで提供したい
- 設定画面を独立したURLで提供したい
- 結果共有用のURLを発行したい (将来)
- PWA化してオフラインでも動作させたい

導入時は **TanStack Router** を推奨する (型安全, 軽量, モダン)。

## 10. エラーハンドリング

### 10.1 エラーの分類と対応

| エラー種別 | 例 | 対応 |
|---|---|---|
| 入力バリデーションエラー | 党名空, 歳入超過 | フォーム側で表示し再入力を促す |
| ロジックエラー | 計算結果の異常値 | clamp等で防御し、必要なら警告表示 |
| 静的データの不整合 | 存在しない政策IDの参照 | 起動時のバリデーションで検出 |
| 予期しないランタイムエラー | 例外発生 | ErrorBoundary でキャッチし、リトライ画面 |

### 10.2 ErrorBoundary

React の Error Boundary を最上位 (App.tsx) に配置し、予期しない例外をキャッチする。

エラー発生時は以下を表示:

- エラーメッセージ (簡潔)
- 「タイトルへ戻る」ボタン
- (デバッグビルドでのみ) スタックトレース

### 10.3 起動時のデータ検証

`data/registry.ts` でレジストリを構築する際に、以下を検証する:

- 各政策・イベントのID重複
- 政策の `sideEffects.targetId` が実在するか
- イベントの `chainedEventIds`, `linkedPolicyId` が実在するか
- トレンドの `counterPolicyIds` が実在するか

検証失敗時は開発者向けに console.error で警告し、本番ビルドでは該当データを無視する。

## 11. パフォーマンス・セキュリティ

### 11.1 パフォーマンス考慮

| 項目 | 対応 |
|---|---|
| 不要な再レンダリング | Zustand のセレクタで必要な値のみ購読 |
| 重い計算 | useMemo で結果キャッシュ。ただし engine の純粋関数は十分軽量 |
| 大きなリスト (政策48本) | 仮想スクロール不要 (規模が小さい) |
| 画像 | MVPは画像不使用. 将来追加時に最適化検討 |
| バンドルサイズ | Tree shaking. 不要な依存を入れない |

非機能要件 (1ターン処理2秒以内, 画面遷移1秒以内, 初期ロード5秒以内) は、上記の方針で容易に達成可能。

### 11.2 セキュリティ

| 項目 | 対応 |
|---|---|
| XSS | React のデフォルト挙動でエスケープ。dangerouslySetInnerHTML は使用しない |
| 個人情報 | 一切収集しない (要件定義書 6.6) |
| 外部リソース | 起動時の外部通信は行わない (静的アセットのみ) |
| 依存ライブラリ | npm audit で脆弱性を定期チェック |

### 12.4 環境変数

MVPでは環境変数は最小限。将来サーバー連携時に拡張。

## 13. 拡張性のための設計指針

### 13.1 機能追加時の設計指針

| 将来機能 | 影響範囲 | 対応方針 |
|---|---|---|
| 教育サポート機能 | UI層 + 新コンポーネント | 既存データの `educationalNote` を読み取って表示するだけ |
| セーブ・ロード機能 | gameStore + 新サービス | localStorage への永続化サービスを追加。State層に閉じる |
| グラフ・チャート可視化 | UI層 | Recharts等を追加。State/Logic層は変更不要 |
| 多言語対応 | データ + UI層 | i18next 等を導入。データの文字列を辞書化 |
| マルチプレイ | State層 + 新サービス | サーバー通信層を追加。Logic層は変更最小 |
| 新シナリオ | データ層のみ | `data/scenarios/` にファイルを追加してレジストリ登録 |
| 新政策・イベント | データ層のみ | 該当カテゴリのファイルに追加 |
| 期間・選挙対象の変更 | GameConfig | Config 値の変更のみ。コードは変更不要 |
| 新しい勝利条件 | engine + データ | WinCondition インターフェースを実装してレジストリ登録 |

### 13.2 アーキテクチャ上の不変原則

将来の変更に対しても以下の原則を守る。

- **レイヤー間の依存方向**: 下位から上位への依存は許さない (Data ← Logic ← State ← UI のみ)
- **純粋関数の維持**: engine 内の関数は副作用を持たない
- **静的データの宣言性**: ロジックの分岐をデータで表現できるものは、コードではなくデータに書く
- **GameState の単一性**: ゲームの状態は1つのオブジェクトで完全に表現する

これらが守られていれば、機能追加時の影響範囲が予測可能になる。
