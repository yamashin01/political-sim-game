# ゲームバランス設計書 — 政治シミュレーションゲーム

機能設計書で構造が確定した計算式・データ項目について、その数値設計の方針・データ記述フォーマット・調整ガイドラインを定義する。本書は**フレームワーク版**であり、各政策・イベント・係数の具体値は本フレームワークに沿って別途確定する。

## 1. 設計方針

### 1.1 フレームワーク先行・具体値後決め

計算式の構造 (機能設計書) と計算係数の数値設計 (本書) は分離する。本書ではまず数値の枠組み (スケール感, 範囲, フォーマット, 関係性) を定め、具体値はプレイテストを経て段階的に確定する。

### 1.2 データ駆動

数値はコードではなくデータ (TypeScriptオブジェクトまたはJSON) として外部化する。これにより、コード変更なしでバランス調整を行える。

### 1.3 段階的な決定とイテレーション

以下の順序で数値を決定する (詳細は9章)。

```
初期 GameState 全体像
  ↓
1ターンあたりの指標変動スケール
  ↓
政策の効果値 (48本)
  ↓
イベントの効果値 (50本)
  ↓
計算係数 (経済情勢補正等)
  ↓
トレンドの進行量
  ↓
プレイテスト → 調整
```

### 1.4 一貫性の維持

同じ重要度の政策は同程度のスケールの効果を持つ。同じ規模のイベントは同程度のスケールの影響を持つ。例外を作る場合は明示的に設計意図を記す。

## 3. 計算係数の数値設計

### 3.1 経済情勢補正係数

選挙時の党のウェイトを算出する式の係数。

```
党のウェイト = 地方支持率 × (1 + 経済情勢補正)

経済情勢補正:
  与党:   +(経済指標 - 50) × α_ruling
  連立党: +(経済指標 - 50) × α_coalition
  野党:   -(経済指標 - 50) × α_opposition
```

| 係数 | 用途 | 想定範囲 | 関係性 |
|---|---|---|---|
| α_ruling | 与党の経済補正の強さ | 0.005 〜 0.02 | 最大値 |
| α_coalition | 連立党の経済補正の強さ | 0.0025 〜 0.01 | α_ruling の約半分 |
| α_opposition | 野党の経済補正の強さ (符号反転) | 0.0025 〜 0.01 | α_ruling の約半分 |

**設計意図**: 経済指標が±50振れたとき、与党のウェイトが±25%程度変動するスケール感を想定。係数を大きくしすぎると経済一辺倒のゲームになり、小さすぎると現職効果が薄れる。

### 3.2 トレンド抑制係数

対抗政策の実行による進行緩和の係数。

```
新しい進行度 = 現在の進行度 + 基準進行量 × 抑制係数

抑制係数 = τ_suppress  (該当する対抗政策が直近5ターン以内に1つ以上実行されている)
        = 1.0          (それ以外)
```

| 係数 | 用途 | 想定範囲 |
|---|---|---|
| τ_suppress | 対抗政策実行時のトレンド進行抑制 | 0.3 〜 0.7 |

**設計意図**: 対抗政策を実行することでトレンド進行を3〜7割に抑えられる程度のスケール。完全停止 (0) では対抗政策一回打つだけで安心してしまい、停止しない (1) ではトレンドに意味がない。

## 4. 政策の数値設計

### 4.1 importance (重要度) のスケール

| 重要度 | 想定 | 必要議席率の例 (乖離0時) |
|---|---|---|
| 1 | 通常法案 | 50% (0.4 + 0.1 × 1) |
| 2 | 中規模法案 | 60% (0.4 + 0.1 × 2) |
| 3 | 大規模・改革法案 | 70% (0.4 + 0.1 × 3) |
| 特殊 | 憲法改正等 | 政策ごとに定義 (例: 0.67 = 2/3) |

48本の重要度配分の目安: 重要度1が多数 (約半数), 重要度2が中程度 (約3分の1), 重要度3が少数 (約5分の1)。

### 4.2 effects (指標への変化量) のスケール

1つの政策が単一の指標に与える変化量の目安。

| 政策の規模 | 単一指標への変化幅 | 例 |
|---|---|---|
| 小規模 | ±1 〜 ±3 | 限定的な制度変更 |
| 中規模 | ±3 〜 ±7 | 重要な制度変更 |
| 大規模 | ±7 〜 ±15 | 抜本的な改革 |

**重要度との関係**: 重要度が高い政策ほど effects の絶対値も大きくなる傾向にあるが、必ずしも比例しない。「重要度は高いが効果は穏やか」「重要度は低いが効果は鋭い」も設計上ありえる。

**指標間の関係 (トレードオフ)**: 1つの政策で複数の指標に効くようにし、必ずトレードオフ (+とー) を持たせる。例えば、消費税減税: 支持率+, 経済+, 財政-。

### 4.3 budgetCost のスケール

政策実行による国家予算への影響 (歳入とは別の財源負担)。

| 規模 | 範囲 |
|---|---|
| 軽い | 0 〜 ±2 |
| 中程度 | ±2 〜 ±5 |
| 重い | ±5 〜 ±10 |

財政赤字 (-) になる政策と財政黒字 (+) になる政策を半々程度に。

### 4.4 政策レコードの記述フォーマット

```yaml
id: P001                              # 一意ID
name: 消費税減税                       # 政策名
category: tax_finance                  # カテゴリ
importance: 2                          # 1〜3
ideologyDirection:                     # 政策のイデオロギー方向
  economic: -1                          # -2〜+2
  social: 0
  diplomatic: 0
timeliness: immediate                   # immediate/short/medium/long
budgetCost: (未定 - 4.3 参照)            # 国家予算への影響
effects:                               # 主要指標への影響 (4.2 参照)
  approval: (未定)
  economy: (未定)
  finance: (未定)
regionalEffects: {}                     # 任意の地方別追加効果
sideEffects: []                         # 任意の副次効果
specialRequirements:                    # 任意 (憲法改正等)
educationalNote: "現実の議論..."         # 一行注釈 (将来用)
```

## 5. イベントの数値設計

### 5.1 scale (規模) のスケール

| scale | 1ターンの発生上限 | 効果のスケール |
|---|---|---|
| small | 3 | ±1 〜 ±5 |
| large | 1 | ±5 〜 ±15 |

### 5.2 effects (指標への変化量) のスケール

| 規模 | 単一指標への変化幅 |
|---|---|
| 小規模 (small) | ±1 〜 ±5 |
| 大規模 (large) | ±5 〜 ±15 |

イベントは政策と異なり、プレイヤーが防げない要素なので、極端な変化を入れる場合は選択肢で部分的に緩和できるようにする。

### 5.3 trigger.baseProbability の範囲

| トリガー種別 | 想定範囲 | 説明 |
|---|---|---|
| scripted | 1.0 (確実発生) | 該当ターンに必ず発生 |
| random | 0.02 〜 0.10 | 完全ランダム発生 |
| conditional | 0.05 〜 0.30 | 状態を満たした上で確率発生 |
| policy_triggered | 0.10 〜 0.40 | 連動政策実行後に確率発生 |
| chained | 1.0 | 連鎖元から確実誘発 |

### 5.4 イベントレコードの記述フォーマット

```yaml
id: E001                                # 一意ID
name: 尖閣諸島周辺での領海侵入            # イベント名
description: "中国海警局の船が..."         # 説明文
category: international_political        # カテゴリ
scale: large                            # small/large
trigger:
  type: random                          # トリガー種別
  baseProbability: (未定 - 5.3 参照)      
affectedRegions: []                     # 任意
immediateEffects: {}                    # 任意 (選択前確定)
choices:
  - id: choice_a
    label: "海保増派・対米連携要請"
    effects:
      approval: (未定)
      diplomacy: (未定)
      # 5.2 参照
  - id: choice_b
    label: "外交ルートで強く抗議のみ"
    effects: (未定)
  - id: choice_c
    label: "静観・冷静対応"
    effects: (未定)
chainedEventIds: []                     # 任意 (連鎖イベント)
educationalNote: "..."                  # 一行注釈
```

## 6. トレンドの数値設計

### 6.1 baseAdvancePerTurn のスケール

毎ターン自動進行する量 (進行度 0〜100 のうちの増分)。

| トレンド | 想定範囲 | 設計意図 |
|---|---|---|
| 緩やかなトレンド | 1 〜 3 | 全期間でゆっくり進行 |
| 中速のトレンド | 3 〜 5 | 期間の中盤で問題化 |
| 急速なトレンド | 5 〜 8 | 早期に高い影響 |

20ターン全体で何 % 進行するかを基準に設計する。例えば baseAdvancePerTurn = 3 なら、20ターンで最大 60、抑制係数 0.5 なら 30 程度になる。

### 6.2 perTurnEffects (進行による毎ターン指標への影響) のスケール

進行度100時の最大影響を基準に設計し、現在の進行度で線形にスケールする。

```
当ターンの影響 = (進行度 / 100) × perTurnEffects
```

| 指標への影響 | perTurnEffects (進行度100時) |
|---|---|
| 軽微 | ±0.5 〜 ±1 |
| 中程度 | ±1 〜 ±3 |
| 深刻 | ±3 〜 ±5 |

進行度50時で半分の影響が出る。

### 6.3 トレンドレコードの記述フォーマット

```yaml
id: aging                               # トレンドID
name: 少子高齢化
description: "..."
baseAdvancePerTurn: (未定 - 6.1 参照)
perTurnEffects:                         # 進行度100時の影響
  finance: (未定)                        # 6.2 参照
  economy: (未定)
counterPolicyIds:                       # 進行を抑制する政策のID
  - P020
  - P021
educationalNote: "..."
```

## 7. 政党・地方ブロック・初期指標の数値設計

### 7.1 政党の初期パラメータ

#### 議席数の配分 (合計465)

| 政党 | モデル | 想定議席比 |
|---|---|---|
| 民自党 | 自民党 | 40〜45% (与党想定で多め) |
| 公明会 | 公明党 | 6〜8% |
| 改革維新党 | 維新の会 | 8〜12% |
| 立憲党 | 立憲民主党 | 15〜20% |
| 人民党 | 共産党+れいわ | 4〜6% |
| プレイヤー党 | (新党) | 20〜25% (野党第一党想定) |

各党の議席数は地方ブロックごとに分配する必要がある。配分パターンは:

- 民自党: 全地方で安定 (地方部やや強)
- 公明会: 都市部やや強
- 改革維新党: 関西で強い (大阪ベース)
- 立憲党: 都市部で強い
- 人民党: 都市部の一部で強い
- プレイヤー党: 開始時はバランス型

#### イデオロギー (3軸)

実在政党のイメージから決める。例:

| 政党 | 経済 | 社会 | 外交 |
|---|---|---|---|
| 民自党 | (未定 - 大きな政府寄り) | (未定 - 保守) | (未定 - 同盟重視) |
| 公明会 | (未定 - 中道〜大きな政府) | (未定 - 中道) | (未定 - 中道) |
| 改革維新党 | (未定 - 小さな政府) | (未定 - 中道〜保守) | (未定 - 中道) |
| 立憲党 | (未定 - 大きな政府寄り) | (未定 - リベラル) | (未定 - 中道) |
| 人民党 | (未定 - 大きな政府) | (未定 - リベラル) | (未定 - 自主路線) |

#### approvalByRegion (地方ごとの支持率)

各党は6地方ブロックそれぞれで初期支持率 0〜100 を持つ。「全国平均」と「地方差」の両面から設計する。

| 政党の特性 | 設計の目安 |
|---|---|
| 全国安定型 | 全地方で同程度の支持率 (例: 全て 30 前後) |
| 地方差大型 | 強い地方は60-70, 弱い地方は10-20 |

#### 初期党資金

| 政党の規模 | 初期党資金 (億円) |
|---|---|
| 与党想定 (民自党) | 大 |
| 中規模党 | 中 |
| 小規模党 / プレイヤー党 | 小 |

具体値はゲーム内のコストバランスから別途決定。

## 8. バランス検証の指標と方法

### 8.1 バランスが取れている状態の定義

以下を満たすことを目標とする。

| 指標 | 目標値 |
|---|---|
| 平均プレイ時間 | 1〜2時間 |
| 平均選挙回数 (10年中) | 2〜4回 |
| 政策通過率 (プレイヤーの提出) | 40〜60% |
| プレイヤーが与党になる確率 | 30〜60% |
| 全エンディングが現実的に到達可能 | はい (失脚エンドも存在感はあるが多発しない) |
| トレンドの最終進行度の幅 | 対抗政策ありで30-50, なしで70-90 |
| 1ターンあたりの指標変動 | 平均 ±3 程度, 最大でも ±15 |

### 8.2 望ましくない状態のシグナル

- **支配戦略の存在**: 特定政策を連発するだけで勝てる → 政策効果のバランス調整
- **死に政策の存在**: 全く選ばれない政策がある → 効果値の見直し or 副次効果の追加
- **エンディングの偏り**: 標準エンディングばかり → 称号判定条件の見直し
- **早期失脚の頻発**: 1〜2年で失脚 → イベント発生確率や効果の調整
- **無風プレイ**: 大きな変化がなく退屈 → イベント発生頻度や効果の調整

### 8.3 検証の方法

| 段階 | 方法 |
|---|---|
| 数値設定直後 | 期待値計算 (平均的なプレイでの指標推移をスプレッドシートでシミュレート) |
| 初期実装後 | 自動プレイ (ランダム選択AIで100回程度プレイし統計を取る) |
| ベータ段階 | 人によるプレイテスト (5〜10名で複数回プレイ) |
| リリース後 | プレイデータ収集 (匿名統計を取れる場合) |

## 9. 数値の段階的な決定プロセス

数値を決める順序のガイドライン。

### Step 1: 全体スケールの決定

最初に「1ターンあたりの指標変動の標準値 (例: ±3 程度) 」を決める。これが全てのスケールの基準となる。

### Step 2: 初期 GameState の決定

- 政党の初期議席数の配分
- 地方の seatShare と totalSeats
- 初期指標の値
- 党のイデオロギー

これにより、ゲーム開始時の状況が確定する。

### Step 3: 政策の効果値の決定

48本の政策について、以下を決定する。

1. 重要度の配分 (1: 約24本, 2: 約16本, 3: 約8本)
2. ideologyDirection (3軸)
3. effects (各指標への変化量)
4. budgetCost
5. sideEffects (必要な政策のみ)
6. specialRequirements (憲法改正等)
7. timeliness

カテゴリごとにまとめて決めるとバランスを取りやすい。

### Step 4: イベントの効果値の決定

50本のイベントについて、以下を決定する。

1. scale (small/large) の決定
2. trigger と baseProbability
3. immediateEffects (該当時)
4. choices と各選択肢の effects
5. chainedEventIds (連鎖イベント)

### Step 5: 計算係数の決定

- 経済情勢補正係数 (α_ruling, α_coalition, α_opposition)
- トレンド抑制係数 (τ_suppress)

これらは政策・イベントの効果値が決まってから、全体バランスを見て決める。

### Step 6: トレンドの進行量・効果の決定

- baseAdvancePerTurn (5本)
- perTurnEffects (各トレンドの進行度100時の影響)
- counterPolicyIds (対応する対抗政策のID)

### Step 7: プレイテストと調整

期待値計算 → 自動プレイ → 人によるプレイテスト の順で検証し、上記の数値を調整する。バランス調整は数回のイテレーションを想定する。

### Step 8: 数値の凍結

ベータテスト終了時点で数値を凍結し、リリースする。リリース後の調整は別バージョンとして管理する。
