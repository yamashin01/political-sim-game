# テスト設計書 — 政治シミュレーションゲーム

本ゲームのテスト方針と最低限実施すべきテストの範囲を定義する。MVPでは「リスクのある場所だけテストする」考え方を採用し、過剰なテストによる開発速度の低下を避ける。

## 1. テスト方針

### 1.1 基本姿勢

- **リスクのある場所だけテストする**: 壊れたときに被害が大きい箇所 (計算ロジック・データ整合性) に集中
- **カバレッジを目標にしない**: 100% カバレッジを追わない。カバレッジは目的でなく副産物
- **テストは開発を速くするためのもの**: 開発を遅くするテストは書かない
- **削除も自由**: 役に立たなくなったテストは削除する

### 1.2 テストが解決する課題

- **計算ロジックのリグレッション防止**: 政策・選挙・スコア計算が意図通りに動き続けることを保証
- **データの整合性**: ID重複や不正参照などのデータ起因のバグを検出
- **仕様の文書化**: テストコードがコードの意図を伝える二次的な文書として機能

### 1.3 テストが解決しない課題

以下はテストでカバーしない。

- 見た目の正しさ (人間の目視で判断)
- ユーザー体験の良し悪し (プレイテストで判断)
- ゲームバランス (プレイテストで判断 — ゲームバランス設計書 8章を参照)
- パフォーマンス (実測で判断)

## 2. テストの対象と対象外

### 2.1 テスト対象 (MVPで実施)

| 対象 | 配置 | 優先度 |
|---|---|---|
| 計算ロジック (engine/) | `tests/engine/` | 高 |
| 静的データの整合性 (data/) | `tests/data/` | 高 |
| 主要な統合フロー (ターン進行・選挙) | `tests/integration/` | 中 (最小限) |

### 2.2 テスト対象外 (MVPでは実施しない)

| 対象 | 理由 |
|---|---|
| UI コンポーネントの個別テスト | テキスト中心UIで、表示は単純なバインディング。バグが出やすい箇所ではない |
| shadcn/ui のコンポーネント | 既にアップストリームでテスト済み |
| React / Zustand / Vite 等のフレームワーク自体 | 既にテスト済み |
| スナップショットテスト | UIが頻繁に変わるため、保守コストの方が大きい |
| E2E テスト (Playwright 等) | セットアップが重く、MVP段階では費用対効果が低い |
| トリビアルなラッパー関数 | テストする価値がない |

将来的に必要となれば追加するが、MVPの初期実装では行わない。

## 3. テストカテゴリ

### 3.1 ユニットテスト (engine 層)

純粋関数のテスト。本ゲームのテストの中心。

**主なテスト対象**:

- イデオロギー乖離・距離の計算
- 必要議席率の計算 (重要度・乖離による変動)
- 選挙ウェイトの算出
- 議席配分 (端数処理含む)
- スコアの計算 (指標平均・党理念達成度・在任比率の合算)
- ターン進行の各フェーズで状態が正しく更新されるか
- トレンド進行 (抑制係数の有無を含む)
- 連立成立判定

**テストの粒度**:

- 1つの関数につき、代表的な入力と境界条件 (0, 100, -2, +2 など) をカバー
- 「全パターン網羅」ではなく「設計意図が壊れていないかの確認」程度

**テスト名の例**:

- 「重要度1, イデオロギー乖離0 の政策で必要議席率が0.5になる」
- 「経済指標100時, 与党のウェイトに正の補正がかかる」
- 「20ターン目を終えるとゲームが終了する」

### 3.2 データ整合性テスト

静的データ (政策48本, イベント50本, トレンド5本等) の整合性を確認する。データを追加するたびに自動で実行されるため、手作業ミスを早期発見できる。

**主なテスト項目**:

- 全データの ID 重複がないこと
- 政策の `sideEffects.targetId` が実在するイベント/政策のIDであること
- イベントの `chainedEventIds`, `trigger.linkedPolicyId` が実在するIDであること
- トレンドの `counterPolicyIds` が実在する政策IDであること
- 列挙値 (カテゴリ・規模等) が有効な値であること
- 必須フィールドが全て埋まっていること
- 数値が想定範囲内にあること (例: importance が 1〜3, ideologyDirection の各軸が -2〜+2)

これらは1つの汎用テスト関数で全データに対して実行できるため、コストが低い割に効果が高い。

### 3.3 統合テスト (最小限)

主要フローを通しでテストする。詳細な検証はせず、「フローが破綻なく完了するか」のみを確認する。

**最小限のシナリオ**:

| シナリオ | 確認内容 |
|---|---|
| ゲーム開始 | 初期GameStateが期待通りに生成される |
| 1ターンを完走 | 全フェーズを通過し、状態が更新される |
| 選挙ターン | 選挙計算が走り、議席が再配分される |
| 20ターンを完走 | エンディングが算出される |
| 失脚エンド | 支持率を意図的に下げると失脚エンドに到達する |

これら以外の統合テストは MVP では行わない。

### 3.4 UI テスト (MVPでは原則行わない)

shadcn/ui のコンポーネントは既にテスト済み、ゲーム固有のコンポーネントはバインディング中心で、UI テストでバグが見つかる可能性は低い。MVP では行わず、人によるプレイテストで補う。

例外: フォーム入力 (党設定画面) で複雑なバリデーションを追加した場合などは、ユニットテストで該当ロジックのみテストする。

## 4. テストの追加判断基準

### 4.1 テストを追加すべきとき

以下のいずれかに該当する場合、テストを追加する。

| 状況 | 理由 |
|---|---|
| バグを発見・修正した | 同じバグの再発を防ぐ (リグレッション防止) |
| 計算ロジックを新規追加・変更する | 仕様通りに動くことを保証する |
| データに新しい整合性ルールを追加する | データ起因のバグを早期発見する |
| 同じ種類のバグが2回以上発生した | パターン化したミスを根本的に防ぐ |
| 複雑な仕様で、コードだけでは意図が読み取りにくい | テストコードを仕様書代わりにする |
| 大きなリファクタリングを行う前 | リファクタが正しいことを確認する足場として |

### 4.2 テストを追加すべきでないとき

以下に該当する場合、テストは追加しない。

| 状況 | 理由 |
|---|---|
| カバレッジを上げるためだけ | テストの目的を見失っている |
| 単純なゲッター/プロパティアクセス | テストする価値がない |
| UIの見た目を確認したいだけ | 目視またはプレイテストで確認すべき |
| 外部ライブラリ (React, Zustand, shadcn/ui) の挙動を確認したいだけ | 既にアップストリームでテスト済み |
| 「念のため」程度の理由 | テストは費用対効果で判断する。「念のため」は判断材料にならない |
| 仕様がまだ流動的 | 仕様が変わるたびにテストも書き直すコストが大きい |

### 4.3 既存のテストを削除すべきとき

以下に該当する場合、既存のテストを削除または更新する。

- テスト対象のコードが削除された
- 仕様変更により、テストが意図通りでなくなった
- メンテナンスコストが、テストが防ぐリスクを上回るようになった
- 同じことを検証する別のテストが追加された

## 5. テストの運用

### 5.1 テストフレームワーク

- **Vitest**: ユニットテスト・統合テスト
- **React Testing Library**: 将来 UI テストが必要となった場合 (MVPでは不使用)

### 5.2 テストファイルの配置

```
tests/
├── engine/        # engine 層のユニットテスト
├── data/          # 静的データの整合性テスト
└── integration/   # 主要フローの統合テスト
```

または、テスト対象のファイルに隣接して `<対象>.test.ts` として配置する形でもよい。チームで配置方針を統一する。

### 5.3 テストの実行

- ローカル: `bun run test` で実行
- 開発中: ファイル変更を監視してリアルタイム実行 (`bun run test --watch`)
- コミット/プッシュ前: 該当箇所のテストが通ることを確認

### 5.4 CI による自動実行 (将来)

GitHub Actions 等で push 時に全テストを実行する。MVP の初期実装では必須ではないが、開発が進んでチーム規模が増えたら導入する。

### 5.5 失敗したテストの扱い

- 失敗したテストはその場で修正するか、テスト自体を削除する
- 「とりあえずスキップ」は禁止 (.skip や .only を残さない)
- リリース時にはすべてのテストが通る状態を保つ

### 5.6 テストの議論

新しいテストを追加するか迷ったときは、本書の 4.1 / 4.2 の基準に照らして判断する。基準に明確に当てはまらない場合は、テストを書かない方を選ぶ (後から必要になれば追加できるが、不要なテストは削除されにくい)。
